Part 1

Database Options for Web Apps

Let's start with the landscape. Every web app needs to store data. The question is where and how.

The Database Landscape

DatabaseTypeBest ForUse In This Course
SQLiteRelational (file)Dev, prototyping, embedded✅ Primary
PostgreSQLRelational (server)Production appsDeployment
MySQLRelational (server)LAMP stacks, WordPressMentioned
MongoDBDocument (NoSQL)Flexible schemas, JSONComparison
RedisKey-Value (in-memory)Caching, sessionsConcepts
We use SQLite for development because it requires zero setup — it's just a file. For production deployments later in the quarter, we'll use PostgreSQL on Azure. The beauty of using an ORM is that switching is usually just a connection string change.

SQLite — Why We Start Here

  • 📦 Zero install — it's just a file on disk
  • ⚡ Fast for development and testing
  • 🔄 Easy to reset — delete the file, re-seed
  • 🚫 No separate server process to manage
  • ⚠️ Limitations: no concurrent writes, limited types
# Your database is literally a file
ls -la prisma/dev.db
# -rw-r--r-- 1 user 28672 Oct 14 dev.db
SQLite stores everything in a single file. This is great for development but not suitable for production with multiple users. It doesn't handle concurrent writes well.

PostgreSQL — Production-Grade

  • Full ACID compliance with concurrent access
  • Rich data types: JSON, arrays, full-text search
  • Scales to millions of rows with proper indexing
  • Available on every cloud platform (Azure, AWS, GCP)
🤖 AI Tip: When asking AI to generate database code, specify "use PostgreSQL syntax" or "use SQLite syntax" — they have differences in type names, auto-increment, and JSON handling.
PostgreSQL is the gold standard for relational databases in web apps. When we deploy to Azure later, we'll switch our Prisma connection from SQLite to PostgreSQL with just one config change.

MongoDB — The NoSQL Option

  • Document-based: stores JSON-like objects (BSON)
  • Schema-flexible: each document can have different fields
  • Great for: content management, IoT, rapid prototyping
  • Not great for: complex relationships, transactions
// MongoDB document — no fixed schema
{
  _id: ObjectId("507f1f77bcf86cd799439011"),
  name: "Laptop",
  price: 999.99,
  specs: { ram: "16GB", storage: "512GB SSD" },
  tags: ["electronics", "computer"]
}
MongoDB is popular but often overused. Many apps that use MongoDB would be better served by a relational database. We'll discuss when NoSQL actually makes sense later in the deck.

Part 2

ORMs & Query Builders

Writing raw SQL for every operation is tedious and error-prone. ORMs provide a layer of abstraction that maps database tables to JavaScript objects.

What Is an ORM?

Object-Relational Mapping — translates between database rows and language objects

Raw SQL

SELECT p.*, c.name AS category
FROM products p
JOIN categories c ON p.category_id = c.id
WHERE p.price > 50;

Prisma ORM

const products = await prisma.product.findMany({
  where: { price: { gt: 50 } },
  include: { category: true }
});
Both produce the same result. The ORM version is more readable and type-safe. However, understanding the underlying SQL is crucial for debugging and performance tuning.

Node.js ORM Landscape

ToolTypeStrengths
PrismaORMType-safe, great DX, schema-first
SequelizeORMMature, feature-rich, model-first
DrizzleORMLightweight, SQL-like API
Knex.jsQuery BuilderFlexible, close to SQL
TypeORMORMDecorators, TypeScript-native

We'll use Prisma in this course — best developer experience and AI tools generate better Prisma code.

Prisma is our choice because it has the best developer experience, generates TypeScript types from your schema, and AI models produce higher-quality Prisma code than other ORMs due to its clear, declarative schema language.

Prisma — The Three Pillars

  1. Prisma Schema — define models in schema.prisma
  2. Prisma Migrate — version-controlled database changes
  3. Prisma Client — auto-generated, type-safe query API
// prisma/schema.prisma
datasource db {
  provider = "sqlite"
  url      = "file:./dev.db"
}

generator client {
  provider = "prisma-client-js"
}

model Product {
  id          Int      @id @default(autoincrement())
  name        String
  price       Float
  createdAt   DateTime @default(now())
  category    Category @relation(fields: [categoryId], references: [id])
  categoryId  Int
}
The Prisma schema is a single source of truth. From it, Prisma generates SQL migrations, TypeScript types, and the query client. This is a schema-first approach — you define what you want, and the tooling generates the rest.

Prisma Workflow

# 1. Initialize Prisma in your project
npx prisma init --datasource-provider sqlite

# 2. Define your models in prisma/schema.prisma
#    (edit the file)

# 3. Create and apply migration
npx prisma migrate dev --name init

# 4. Generate the client
npx prisma generate

# 5. Explore your data visually
npx prisma studio
🤖 AI Tip: "Generate a Prisma schema for an e-commerce app with products, categories, and orders" is a great starting prompt — but always review the relationships!
Walk through each step. Prisma Studio is excellent for visual data exploration. Students should run prisma studio to verify their schema and seed data.

Part 3

AI-Generated Database Schemas

Evaluating What the AI Produces

AI is great at generating initial schemas, but you need to critically evaluate the output. Let's look at what to check.

AI Schema Checklist ✅

  • ✅ Are all relationships defined correctly? (1:1, 1:N, N:M)
  • ✅ Are required fields marked as non-nullable?
  • ✅ Is @unique applied where appropriate (email, slug)?
  • ✅ Are createdAt / updatedAt timestamps included?
  • ✅ Does cascade delete make sense for each relation?
  • ✅ Are indexes defined for frequently queried fields?
  • ✅ Are decimal types used for money (not Float)?
Go through each item. The most common AI mistake is using Float for money — Prisma supports Decimal type. Another common mistake is missing cascade delete rules, which can leave orphaned records.

Common AI Schema Mistakes

❌ AI often generates: 
model Order {
  id       Int    @id @default(autoincrement())
  total    Float  // ❌ Float for money!
  status   String // ❌ No enum, any string accepted
  userId   Int    // ❌ No @relation defined
}
✅ Better version: 
model Order {
  id        Int         @id @default(autoincrement())
  total     Decimal     // ✅ Precise money type
  status    OrderStatus @default(PENDING) // ✅ Enum
  user      User        @relation(fields: [userId], references: [id])
  userId    Int
  createdAt DateTime    @default(now())
}

enum OrderStatus {
  PENDING
  PROCESSING
  SHIPPED
  DELIVERED
  CANCELLED
}
Show both versions side-by-side. Ask students why Float is bad for money (floating point precision errors — 0.1 + 0.2 !== 0.3). Enums prevent invalid status values from being saved.

Part 4

Migrations & Seed Data

Once you have a schema, you need to create the actual database tables and populate them with test data.

Database Migrations

  • Version-controlled changes to your database schema
  • Each migration is a SQL file that modifies the database
  • Applied in order — can be rolled back
  • Critical for team collaboration and deployment
# Create a migration after schema changes
npx prisma migrate dev --name add_order_items

# This generates:
# prisma/migrations/20241014_add_order_items/migration.sql

# Apply migrations in production
npx prisma migrate deploy
Migrations are like Git commits for your database schema. Never modify a migration after it's been applied. Always create a new migration for changes.

Seed Data with AI

// prisma/seed.js
const { PrismaClient } = require('@prisma/client');
const prisma = new PrismaClient();

async function main() {
  // Clear existing data
  await prisma.orderItem.deleteMany();
  await prisma.order.deleteMany();
  await prisma.product.deleteMany();
  await prisma.category.deleteMany();

  const electronics = await prisma.category.create({
    data: {
      name: 'Electronics',
      products: {
        create: [
          { name: 'Laptop', price: 999.99, stock: 50 },
          { name: 'Headphones', price: 79.99, stock: 200 },
        ]
      }
    }
  });
  console.log(`Seeded ${electronics.name} with products`);
}
main().catch(console.error).finally(() => prisma.$disconnect());
Show how to use nested creates in Prisma for seed data. The delete order matters — you must delete in reverse dependency order. AI tools are great at generating realistic seed data, but always verify the relationships.

Running Seeds

// package.json
{
  "prisma": {
    "seed": "node prisma/seed.js"
  }
}
# Run the seed script
npx prisma db seed

# Or reset everything and re-seed
npx prisma migrate reset
🤖 AI Prompt: "Generate Prisma seed data for an e-commerce app with 5 categories, 20 products, 10 users, and 15 orders with order items. Use realistic product names and prices."
The prisma.seed config in package.json tells Prisma which script to run. The migrate reset command drops the database, re-applies all migrations, and runs the seed script — very useful during development.

Part 5

SQL vs NoSQL

When to Use Each

This is one of the most common questions in web dev. The answer, as always, is "it depends."

SQL vs NoSQL Decision Guide

FactorSQL (PostgreSQL)NoSQL (MongoDB)
Data StructureFixed, predictableFlexible, evolving
RelationshipsStrong (JOINs)Weak (embedded/refs)
TransactionsFull ACIDLimited
ScalingVertical firstHorizontal
Schema ChangesMigration requiredJust write new fields
Query PowerComplex JOINs, aggregationsGood for document lookups

Rule of thumb: If you're building a business app with clear entities and relationships → SQL

Most web applications should start with SQL. NoSQL is great for specific use cases like logging, content management, or IoT data. If you're asking "should I use NoSQL?" the answer is probably no — unless you have a specific reason.

The NoSQL Trap 🪤

⚠️ Common Mistake: AI tools often suggest MongoDB for projects where PostgreSQL would be better. Watch out for:
  • Data duplication instead of relationships
  • No referential integrity — orphaned records
  • "Flexible schema" becoming "inconsistent data"
  • Complex aggregation pipelines that a JOIN would solve

Use NoSQL when: truly schema-less data, massive write throughput, document-centric access patterns, or caching layers

Many tutorial sites and AI tools default to MongoDB because it's "easier to get started." But the long-term cost of denormalized data in a relational domain is high. Always think about your data relationships first.

Part 6

React State Management

From Component State to Global State

Now let's shift to the frontend. We covered useState in Week 3. Now we'll look at managing state that needs to be shared across many components.

State Management Spectrum

Simple ◄──────────────────────────────────► Complex

useState    useReducer    Context API    Zustand    Redux
  (local)     (local)      (global)     (global)   (global)
   └─ 1 component  └─ complex logic  └─ theme, auth  └─ large apps
  • useState — simple values in one component
  • useReducer — complex state logic (like a mini Redux)
  • Context API — share state across component tree (built-in)
  • Zustand — lightweight external store (our recommendation)
  • Redux — heavy-duty, enterprise-scale state
Don't reach for Redux until you need it. Most apps do fine with Context or Zustand. We'll focus on Context API tonight since it's built into React.

Context API — Creating a Context

// context/CartContext.jsx
import { createContext, useContext, useReducer } from 'react';

const CartContext = createContext();

function cartReducer(state, action) {
  switch (action.type) {
    case 'ADD_ITEM':
      return { ...state, items: [...state.items, action.payload] };
    case 'REMOVE_ITEM':
      return {
        ...state,
        items: state.items.filter(item => item.id !== action.payload)
      };
    case 'CLEAR':
      return { ...state, items: [] };
    default:
      throw new Error(`Unknown action: ${action.type}`);
  }
}
We combine Context with useReducer for predictable state updates. The reducer is a pure function — given the same state and action, it always produces the same result. This pattern will feel familiar if students have used Redux before.

Context API — Provider & Hook

// context/CartContext.jsx (continued)
export function CartProvider({ children }) {
  const [state, dispatch] = useReducer(cartReducer, { items: [] });

  const addItem = (product) =>
    dispatch({ type: 'ADD_ITEM', payload: product });
  const removeItem = (id) =>
    dispatch({ type: 'REMOVE_ITEM', payload: id });
  const clearCart = () =>
    dispatch({ type: 'CLEAR' });

  return (
    <CartContext.Provider value={{ ...state, addItem, removeItem, clearCart }}>
      {children}
    </CartContext.Provider>
  );
}

// Custom hook for easy access
export function useCart() {
  const context = useContext(CartContext);
  if (!context) throw new Error('useCart must be used within CartProvider');
  return context;
}
The custom hook pattern is important — it provides a clean API and catches the common mistake of using the context outside its provider. Wrap your App with CartProvider, then call useCart() from any component.

Using the Context in Components

// components/ProductCard.jsx
import { useCart } from '../context/CartContext';

function ProductCard({ product }) {
  const { addItem } = useCart();

  return (
    <div className="product-card">
      <h3>{product.name}</h3>
      <p>${product.price}</p>
      <button onClick={() => addItem(product)}>
        Add to Cart
      </button>
    </div>
  );
}

// components/CartBadge.jsx
function CartBadge() {
  const { items } = useCart();
  return <span className="badge">{items.length}</span>;
}
Both components access the same cart state without prop drilling. The ProductCard adds items, and the CartBadge shows the count — they don't need to know about each other.

Zustand — The Lightweight Alternative

// store/cartStore.js
import { create } from 'zustand';

const useCartStore = create((set) => ({
  items: [],
  addItem: (product) =>
    set((state) => ({ items: [...state.items, product] })),
  removeItem: (id) =>
    set((state) => ({
      items: state.items.filter(item => item.id !== id)
    })),
  clearCart: () => set({ items: [] }),
  get totalPrice() {
    return this.items.reduce((sum, item) => sum + item.price, 0);
  }
}));

// Usage — no Provider needed!
function CartBadge() {
  const itemCount = useCartStore((state) => state.items.length);
  return <span>{itemCount}</span>;
}
Zustand is simpler than Context + useReducer. No Provider wrapping, no boilerplate. It's our recommended choice for projects beyond the basics. Notice how you can select specific slices of state to prevent unnecessary re-renders.

Caching Strategies

Caching is how we make apps fast. Let's look at the different layers where caching happens.

Caching Layers

User Request
    │
    ▼
┌─────────────────┐
│  Browser Cache   │  HTTP Cache-Control headers
├─────────────────┤
│  CDN Cache       │  Static assets, API responses
├─────────────────┤
│  App Memory      │  In-process cache (Node.js Map)
├─────────────────┤
│  Redis / Memcache│  Shared cache across servers
├─────────────────┤
│  Database Cache  │  Query cache, connection pool
└─────────────────┘
Each layer serves a different purpose. Browser cache prevents network requests entirely. Redis sits between your app and database, storing frequently accessed data in memory. We'll focus on browser cache and in-memory cache tonight.

Practical Caching in Express

// Simple in-memory cache
const cache = new Map();

app.get('/api/products', async (req, res) => {
  const cacheKey = 'all-products';

  // Check cache first
  if (cache.has(cacheKey)) {
    return res.json(cache.get(cacheKey));
  }

  // Cache miss — query database
  const products = await prisma.product.findMany({
    include: { category: true }
  });

  // Store in cache with TTL
  cache.set(cacheKey, products);
  setTimeout(() => cache.delete(cacheKey), 60000); // 1 min TTL

  res.json(products);
});
⚠️ Note: In-memory cache is lost on server restart and not shared across instances. Use Redis for production caching.
This is a basic caching pattern. The tricky part is cache invalidation — when data changes, you need to clear the cache. The joke is that there are only two hard problems in CS: cache invalidation, naming things, and off-by-one errors.

Part 8

Common AI Pitfalls

What AI Gets Wrong with Databases

This is critical. AI tools are great at generating database code, but they have consistent blind spots. Let's learn to spot them.

Pitfall 1: The N+1 Query Problem

❌ AI often generates: 
// Fetches all orders, then queries user for EACH order
const orders = await prisma.order.findMany();
for (const order of orders) {
  order.user = await prisma.user.findUnique({
    where: { id: order.userId }
  });
  // 1 query for orders + N queries for users = N+1 queries!
}
✅ Fix — use include (eager loading): 
const orders = await prisma.order.findMany({
  include: { user: true }
});
// Just 1 query with a JOIN! 🚀
The N+1 problem is the #1 performance issue in ORM-based apps. If you have 100 orders, the bad version makes 101 database queries. The fixed version makes just 1. AI generates the bad version surprisingly often because it writes procedural-style code.

Pitfall 2: Missing Indexes

❌ No index on frequently queried field: 
model Product {
  id         Int    @id @default(autoincrement())
  name       String
  categoryId Int    // Queried often, but no index!
  price      Float
}
✅ Add indexes for query performance: 
model Product {
  id         Int      @id @default(autoincrement())
  name       String
  categoryId Int
  price      Float
  category   Category @relation(fields: [categoryId], references: [id])

  @@index([categoryId])
  @@index([price])
}
Without an index, the database does a full table scan for every query on that field. AI tools rarely add indexes unless you explicitly ask. For any field used in WHERE, ORDER BY, or JOIN — add an index.

Pitfall 3: Improper Relationships

❌ Storing related data as comma-separated string: 
model Product {
  id    Int    @id @default(autoincrement())
  name  String
  tags  String // "electronics,sale,featured" 😱
}
✅ Use a proper many-to-many relationship: 
model Product {
  id   Int   @id @default(autoincrement())
  name String
  tags Tag[]
}

model Tag {
  id       Int       @id @default(autoincrement())
  name     String    @unique
  products Product[]
}
This is a surprisingly common AI output, especially when asked to "add tags to products." A comma-separated string means you can't query by tag, can't enforce uniqueness, and can't count products per tag efficiently.

Pitfall 4: No Connection Pooling

❌ Creating a new client per request: 
app.get('/api/products', async (req, res) => {
  const prisma = new PrismaClient(); // ❌ New connection each request!
  const products = await prisma.product.findMany();
  res.json(products);
  // Connection never closed — memory leak!
});
✅ Single shared client instance: 
// Create ONE client at app startup
const prisma = new PrismaClient();

app.get('/api/products', async (req, res) => {
  const products = await prisma.product.findMany();
  res.json(products);
});

// Graceful shutdown
process.on('SIGTERM', async () => {
  await prisma.$disconnect();
});
Databases have a limited number of connections. Creating a new client per request will exhaust the connection pool and crash your app under load. Prisma handles connection pooling internally — you just need one client instance.

Part 9

Data Validation

Client + Server

Data validation must happen on BOTH the client and server. Client-side for UX, server-side for security. Never trust data from the browser.

Validation Strategy

Browser (UX)           Server (Security)         Database (Integrity)
┌──────────────┐      ┌──────────────────┐      ┌─────────────────┐
│ HTML5 attrs  │      │ Zod / Joi schema │      │ NOT NULL        │
│ React forms  │ ───► │ Express middleware│ ───► │ UNIQUE          │
│ Instant      │      │ Sanitize input   │      │ CHECK           │
│ feedback     │      │ Reject bad data  │      │ Foreign keys    │
└──────────────┘      └──────────────────┘      └─────────────────┘

All three layers are needed. Never rely on just one.
The browser layer can be bypassed with curl or Postman. The database layer is your last line of defense but gives bad error messages. The server layer is where most of your validation logic should live.

Server Validation with Zod

import { z } from 'zod';

const ProductSchema = z.object({
  name: z.string().min(2).max(100),
  price: z.number().positive().max(99999),
  categoryId: z.number().int().positive(),
  description: z.string().max(1000).optional(),
});

app.post('/api/products', async (req, res) => {
  // Validate request body
  const result = ProductSchema.safeParse(req.body);

  if (!result.success) {
    return res.status(400).json({
      error: 'Validation failed',
      details: result.error.issues
    });
  }

  const product = await prisma.product.create({
    data: result.data  // Only validated data reaches the DB
  });
  res.status(201).json(product);
});
Zod gives you runtime type checking in JavaScript. The safeParse method returns either success with validated data, or failure with descriptive error messages. This is the recommended pattern for all API endpoints.

SQL Injection — Still a Threat

❌ AI sometimes generates raw queries: 
// NEVER do this!
app.get('/api/search', async (req, res) => {
  const { query } = req.query;
  const results = await prisma.$queryRawUnsafe(
    `SELECT * FROM products WHERE name LIKE '%${query}%'`
  );
  // Input: ' OR 1=1 -- 💀
});
✅ Use parameterized queries or the ORM: 
// Option 1: Prisma ORM (preferred)
const results = await prisma.product.findMany({
  where: { name: { contains: query } }
});

// Option 2: Parameterized raw query
const results = await prisma.$queryRaw`
  SELECT * FROM products WHERE name LIKE ${'%' + query + '%'}
`;
SQL injection is one of the oldest web vulnerabilities and it's still in the OWASP Top 10. AI-generated code sometimes uses raw queries when the ORM API would work fine. Always prefer the ORM. If you must use raw SQL, always use parameterized queries.

Key Takeaways

  1. Use SQLite for dev, PostgreSQL for production
  2. Prisma = schema-first ORM with great DX
  3. Always review AI-generated schemas for relationships & types
  4. Watch for N+1 queries — use include
  5. Context API for global state, Zustand for more
  6. Validate on client (UX) and server (security)
  7. Never trust AI with raw SQL — check for injection
Summarize the key points before the exercise. Emphasize that AI is a great starting point for database code, but critical review is essential.
